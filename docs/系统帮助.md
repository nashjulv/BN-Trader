# BN-Trader v1.0.0 系统帮助

> 在应用内按 `F1` 可随时打开可搜索的帮助中心。

## 1. 欢迎使用

了解系统定位、使用边界与推荐操作顺序。

### 系统能做什么

BN-Trader 将市场行情、交易策略、资金状态和风险规则放在同一个桌面工作台中。系统可以辅助判断与执行，但不会消除市场风险。

### 推荐使用顺序

1. 连接或跳过 API，先进入主界面熟悉行情数据。
2. 设置资金、风控和持仓限制。
3. 配置策略参数并保存。
4. 先用手动或模拟方式验证，再考虑自动交易。
5. 定期查看日志并完成复盘。

## 2. 首次配置

完成 API、资金、风控和策略的首次设置。

### API 设置

1. 打开左侧“设置”。
2. 填入 Binance API Key 与 Secret Key。
3. 点击连接测试，确认密钥、网络、账户权限和时间同步均正常。
4. 保存后返回主界面并检查右下角 API 状态。

> 提示：建议创建专用 API，仅开启所需交易权限，绝不要开启提现权限。

### 全局参数

依次设置资金、风控和持仓参数。单笔风险、日亏损上限与最大回撤应以个人可承受损失为基准，而不是以期望收益为基准。

### 策略参数

在策略页选择适用的行情类型并保存参数。启用自动交易前，系统会检查策略是否已完成配置。

## 3. 主界面

认识行情、资金、图表、日志和下单区域。

### 顶部状态栏

显示交易模式、交易对、当前价格、今日盈亏、总权益、胜率、主题切换与快捷设置入口。

### 左侧状态区

集中展示当前行情场景、资金概览、风险状态和持仓摘要。

### 图表与日志

中间区域展示 K 线、均线与成交量；下方日志记录交易、风控与系统事件。

### 手动下单

右侧下单面板支持交易对、订单类型、价格、数量、止损与止盈。提交前会执行风控检查并要求确认。

## 4. 自动化投资任务台

先决定哪些币种参与投资及其预算，再为币种配置策略任务，并由客户手动启动。

### 为什么资金分配与任务分开

资金分配表回答“每个币种最多可以使用多少总资金”，任务表回答“何时、按什么策略评估和执行”。一个币种可以有多个任务，但它们共享同一个币种预算，系统会扣除该币种全部自动持仓已经占用的资金，防止多个任务重复获得完整预算。

### 设置参与投资的币种

1. 打开左侧“自动”进入自动化投资任务台。
2. 在“币种资金分配”中新增币种；可选择预设交易对，也可输入币安现货 USDT 交易对，例如 ADAUSDT。
3. 填写该币种占总资金的分配比例，并选择是否参与自动化投资。
4. 确认所有启用币种的比例合计不超过 100%；未分配部分保持为空闲资金，不会被自动任务主动占用。
5. 需要暂时退出某个币种时优先停用；删除币种会同时删除其关联任务，并要求先停止运行中的任务。

> 提示：分配比例是预算上限，不代表系统会立即买入对应金额。实际订单仍取决于策略信号、场景额度、可用资金和全部风控规则。

### 新增和启动自动化任务

1. 先在资金分配表中新增并启用交易对。
2. 新增任务并从已参与投资的币种中选择交易对。
3. 选择 K 线周期、自动匹配或固定策略、允许方向、单次使用币种预算比例、评估周期和每日最大执行次数。
4. 保存后复核资金、策略和 API 环境，再手动启动选中任务或全部已启用任务。
5. 通过执行记录查看行情获取、场景识别、策略信号与风控拦截原因。

### 停用、停止与重启安全

- 停用币种会停止该币种的任务，但保留资金比例与任务设置，便于以后重新启用。
- 停止任务只停止调度，不会删除配置，也不会绕过已有持仓的止损和风险管理。
- 关闭或升级应用后，资金分配与任务配置仍会保留。
- 为防止无人确认自动恢复实盘，应用每次启动后所有任务都保持停止，必须由客户再次手动启动。

### USDC/USDT 的特殊风险

USDC/USDT 通常围绕 1 附近波动，但稳定币并不等于无风险。脱锚、流动性变化、手续费和最小价格跳动都可能使窄区间策略失效。默认使用震荡策略和较低比例，只是审慎初始值，不是收益承诺；出现明显脱锚时应停止区间逻辑并优先保护本金。

## 5. 策略设计原理

理解交易策略如何从市场假设转化为可验证、可执行的规则。

### 策略不是预测，而是一组条件

成熟的策略不要求每次预测正确，而是在特定市场条件下，用一致的入场、退出和仓位规则重复执行。策略价值来自长期样本中的正向期望，而不是某一笔交易的输赢。

### 完整策略的五个组成部分

- 适用环境：策略在哪类行情、交易对和周期中有效。
- 入场条件：哪些可以观察和复现的信号允许开仓。
- 失效条件：出现什么情况说明原有判断不再成立。
- 退出规则：止损、止盈、移动止损和时间退出如何配合。
- 仓位规则：每笔允许承担多少风险，连续亏损后如何降仓。

### 五类场景与策略思路

- 趋势行情：顺着主要方向参与，重点防止追涨杀跌，并用移动止损保护已有利润。
- 震荡行情：围绕区间边界寻找机会，一旦有效突破区间，应停止原有高抛低吸逻辑。
- 突破行情：关注价格突破与成交量确认，防范假突破，避免在确认不足时扩大仓位。
- 反转行情：属于逆势交易，信号失败概率通常更高，应降低仓位并使用更严格的失效条件。
- 极端行情：流动性和波动可能失常，观望、减仓和保护本金通常比寻找机会更重要。

### 期望值比胜率更重要

策略期望值可以简单理解为：胜率 × 平均盈利 − 败率 × 平均亏损。高胜率策略仍可能因少数大额亏损而失效；胜率较低的策略也可能依靠较好的盈亏比获得正向期望。因此应同时观察胜率、盈亏比、最大回撤和样本数量。

### 参数设计与过度拟合

1. 先写清楚市场假设，再选择能够表达该假设的少量指标。
2. 为参数保留合理区间，不追求历史数据中的唯一最佳数字。
3. 使用未参与调参的数据验证策略，并覆盖不同市场阶段。
4. 将手续费、滑点、延迟和无法成交纳入评估。
5. 上线后用小仓位观察，只有行为与预期一致时才逐步扩大。

> 提示：历史回测表现不代表未来收益。参数越复杂，越容易只记住历史噪声。

### 策略失效与停止规则

每个策略都应预先定义暂停条件，例如回撤超过阈值、连续亏损超出历史分布、成交质量明显下降或市场结构发生变化。暂停是策略管理的一部分，不应等同于失败。

## 6. 术语、指标与默认值

理解 K 线、MA、MACD、RSI、ATR、ADX 等术语，以及系统为何采用当前初始参数。

### 先理解“默认值”

系统默认参数是用于启动、演示和建立统一测试基线的通用值，不是针对某个交易对优化出的最佳参数，也不代表收益承诺。周期数字表示 K 线根数：在默认 15 分钟周期中，20 根 K 线约覆盖 5 小时；切换周期后，同一个参数代表的真实时间也会改变。

> 提示：修改任何参数前，应先明确交易周期、市场假设和可承受风险，并用独立数据验证。不要因为短期回测更好而频繁调参。

### K 线与 OHLCV

- K 线：在一个时间周期内汇总价格变化的数据单元。
- Open / 开盘价：该周期第一笔记录价格。
- High / 最高价：该周期内出现的最高价格。
- Low / 最低价：该周期内出现的最低价格。
- Close / 收盘价：该周期最后一笔记录价格，系统多数指标以它为输入。
- Volume / 成交量：该周期的成交规模，用于判断价格运动是否得到交易活跃度支持。
- Timeframe / 周期：每根 K 线覆盖的时间。系统默认 15m，是为了在短线响应速度与噪声之间取得初始平衡。

### MA、SMA 与 EMA

MA 是移动平均线的统称，用于平滑价格并观察方向。SMA 对窗口内每个价格赋予相同权重；EMA 对最近价格赋予更高权重，因此反应更快，但也更容易受到短期噪声影响。金叉是快线上穿慢线，死叉是快线下穿慢线；它们是滞后确认信号，不等同于必然上涨或下跌。

- 趋势策略默认 EMA 5：代表较快的短期价格反应。
- 趋势策略默认 EMA 20：代表较稳定的短期基准，也与约一个常用交易观察窗口相匹配。
- 采用 5/20 的原因：让快慢线之间有足够差异，便于观察短线方向变化，同时避免使用极短周期造成过多交叉。
- 图表还计算 SMA 5/10/20/60，用于从短到中期观察价格结构；这些均线主要用于展示和场景分析，并非每一条都会直接触发订单。

### MACD（12、26、9）

MACD 用两条不同速度的 EMA 衡量动量变化。MACD 线通常是 EMA12 减 EMA26；信号线是 MACD 线的 9 周期 EMA；柱状图是两者的差。柱状图扩大表示动量差增强，收缩表示动量差减弱，但不能单独证明趋势反转。

- 12：较快 EMA 周期，用于感知近期变化。
- 26：较慢 EMA 周期，用作中短期参照。
- 9：信号线平滑周期，用于减少 MACD 线的短期抖动。
- 选择 12/26/9 的原因：这是广泛使用的基准组合，便于客户理解和跨平台核对，也在灵敏度与稳定性之间提供合理起点。
- 当前趋势策略只有在 EMA 交叉、价格位于慢线同方向一侧、MACD 柱状图方向一致时才生成信号，MACD 在这里用于确认而不是单独下单。

### ADX（14，参考阈值 25）

ADX 衡量趋势强度而不表示方向，数值越高通常代表方向性越明显；方向要结合 +DI、-DI 或价格结构判断。14 周期用于平衡反应速度与稳定性，25 常被用作趋势与非趋势环境的参考分界，但不同资产和周期需要重新验证。

> 提示：当前系统会计算 ADX14 用于场景强度判断；策略设置页的“ADX 阈值 25”目前是预留参数，尚未在趋势策略的入场条件中直接拦截信号。

### RSI（14、30/70 与 25/75）

RSI 是 0 到 100 的动量振荡指标。较低数值表示近期下跌动量较强，较高数值表示近期上涨动量较强。“超卖”不保证马上上涨，“超买”也不保证马上下跌；强趋势中 RSI 可能长时间停留在极端区域。

- 14 周期：常见基准，兼顾短线灵敏度和稳定性。
- 震荡策略使用 30/70：与布林带边界共同确认区间内的相对极端位置。
- 反转策略使用更严格的 25/75：减少普通波动被误判为反转的次数。
- 反转策略还要求 10 根 K 线内出现至少 5 点 RSI 背离差值，以提高信号选择性。

### 布林带（20，2 倍标准差）

布林带由 20 周期中轨和上下波动带组成。中轨是 SMA20，上下轨距离为 2 倍标准差，用于描述近期价格相对位置和波动扩张。价格触轨不等于一定反转；趋势行情中价格可能沿着上轨或下轨持续运行。

- 20 周期：提供较稳定的近期价格基准，并与系统的 20 根回看窗口保持一致。
- 2 倍标准差：用于过滤部分普通波动，是常用研究起点，但不是价格必然留在带内的概率保证。
- 当前震荡策略要求价格接近上下轨并同时满足 RSI 30/70，避免仅凭一次触轨下单。

### ATR（14）与波动倍数

ATR 是平均真实波幅，用于衡量价格实际波动大小，不表示涨跌方向。它同时考虑当根高低差和相对前收盘价的跳空，因此比简单振幅更适合估计止损距离与异常波动。

- 14 周期：常见波动观察窗口，避免只依据最近一两根 K 线。
- 趋势策略优先使用 2×ATR 设置结构化止损；无法计算 ATR 时才回退到默认 2% 止损。
- 突破页初始 ATR 过滤倍数为 1.5，用意是要求突破幅度明显高于普通波动。
- 极端策略初始 ATR 倍数为 3.0，用意是只对显著偏离常态的单根价格变化作出反应。

> 提示：突破策略目前会计算突破强度，但“ATR 过滤 1.5 倍”尚未直接参与最终信号条件。

### 成交量倍数与回看周期

- 成交量倍数：当前成交量除以历史平均成交量。1.5 倍表示当前量能约为基准的 150%。
- 突破策略默认回看 20 根 K 线，用这段区间的高低点作为阻力和支撑参考。
- 突破成交量默认 1.5 倍，用于减少缺乏参与度的假突破。
- 突破阈值默认 0.8%，用于要求价格越过历史边界，而不是仅仅触碰边界。
- 极端行情的成交量激增默认 3 倍，用于描述非常态活跃度，但该参数目前为预留配置，尚未直接参与极端策略信号。

### 支撑、阻力、突破与背离

- 支撑：历史上买方力量曾增强、价格停止下跌的区域，不是一条绝对价格线。
- 阻力：历史上卖方力量曾增强、价格停止上涨的区域。
- 突破：价格有效越过支撑或阻力；成交量和波动确认可以降低但不能消除假突破。
- 背离：价格创新高或新低，但动量指标未同步创新高或新低，表示原趋势动量可能减弱，不代表反转必然发生。

### 止损、止盈、移动止损与盈亏比

- 止损：在交易假设失效时限制损失的退出条件。
- 止盈：达到预定目标后退出，不代表目标价格一定会成交。
- 移动止损：随着有利价格方向移动保护利润，不应在价格不利时反向放宽。
- 盈亏比 2.0：计划潜在盈利约为计划风险的 2 倍。突破策略以突破位置为止损基础，并据此计算目标价。
- 趋势初始止损 2%、止盈 5%、移动止损 1.5% 是用于建立测试基线的风险参数，并非对所有币种都合适。

> 提示：当前趋势策略的移动止损参数尚未接入持仓动态更新；震荡策略的止盈会优先使用布林带中轨并确保最低计划盈亏比；突破策略的百分比止损目前由突破位止损替代。

### 为什么默认启用三类策略

趋势、震荡和突破策略默认启用，是因为它们依赖相对直观、可观察的价格结构，并采用多个条件共同确认。反转与极端策略默认关闭，是因为逆势和异常行情中的噪声、滑点与执行风险更高，需要客户主动理解并验证后再开启。

- 趋势：默认启用，使用 EMA 交叉、价格位置与 MACD 动量共同确认。
- 震荡：默认启用，使用布林带与 RSI 双重确认。
- 突破：默认启用，使用历史边界、0.8% 突破距离与 1.5 倍成交量确认。
- 反转：默认关闭，信号置信度和容错更低，逆势交易风险较高。
- 极端：默认关闭，高波动时滑点和流动性可能显著恶化，系统更倾向于观望。

### 为什么数据不足时显示“震荡”

场景识别至少需要足够的历史 K 线才能计算稳定指标。当数据不足时，系统暂以“震荡行情、置信度 0.5、建议仓位 10%”作为中性保守占位状态。它表示“尚无足够证据确认趋势或突破”，不表示系统已经判断市场必然横盘。

> 提示：看到数据不足的初始状态时，应等待数据完成加载，不应仅根据该占位状态执行交易。

## 7. 风险管理

理解风险预算、仓位、止损、回撤及系统交易拦截的原理。

### 风险控制的目标

风险控制不是避免所有亏损，而是让单次错误、连续错误和极端行情都不会轻易终止整个交易计划。先确保能够留在市场中，才有机会让策略的长期期望发挥作用。

### 先确定风险金额，再计算仓位

单笔风险金额通常由账户权益乘以单笔风险比例得到。仓位数量可用“风险金额 ÷ 入场价与止损价的距离”估算。止损距离越大，可承受的仓位就应越小；不能先决定仓位，再为了容纳仓位而随意放宽止损。

### 止损代表交易假设失效

止损应放在能够证明原有交易逻辑不再成立的位置，并在下单前确定。过近的止损容易被正常波动触发，过远的止损会放大损失；应结合市场结构、波动率、手续费和滑点设置。

### 风险收益比不是收益保证

风险收益比描述潜在盈利与潜在亏损的关系，但目标价并不保证成交。评估时需要与实际胜率共同考虑，并用长期成交记录验证平均盈利和平均亏损，而不是只看下单时填写的数字。

### 三层风险限制

- 单笔：最大亏损、最大盈利、最长持仓时间与仓位占比。
- 日度：交易次数、累计亏损、连续亏损与冷静期。
- 账户：最大回撤、最低储备金比例与保证金约束。

### 为什么要管理连续亏损

连续亏损可能只是概率波动，也可能表示策略与当前市场不匹配。系统通过降低可交易资金、进入冷静期或触发复盘，防止情绪化加仓和试图立即追回损失。

### 理解最大回撤

最大回撤衡量账户从历史高点到随后低点的最大跌幅。亏损后的恢复并不对称：亏损 20% 需要盈利 25% 才能回到原点，亏损 50% 则需要盈利 100%。因此控制深度回撤比追求短期高收益更重要。

### 相关性与总风险

同时持有多个高度相关的交易对，并不等于真正分散风险。它们可能在同一市场冲击下同时亏损。评估账户风险时，应把方向相同、驱动因素相近的仓位视为一组风险暴露。

### 风险拦截

当订单不符合当前规则时，系统会阻止提交并说明原因。请修改订单或风险参数，不要通过扩大限额来绕过警告。

### 强制复盘

达到连续亏损等触发条件后，系统会暂停交易并要求完成复盘。只有满足恢复条件后，交易权限才会重新开放。

### 客户自检清单

1. 这笔交易的入场依据和失效条件是否清楚？
2. 触发止损时的金额是否在可承受范围内？
3. 现有相关仓位是否会放大同一方向的风险？
4. 当前是否处于连续亏损、疲劳或情绪激动状态？
5. 最坏情况下的滑点和无法及时退出是否仍可接受？

> 提示：任何无法清楚回答上述问题的交易，都更适合暂缓，而不是降低风控标准。

## 8. 交易操作

安全地使用手动和自动交易模式。

### 手动模式

1. 确认交易对与当前价格。
2. 选择限价或市价单并输入数量。
3. 设置止损与止盈。
4. 阅读确认信息后再提交订单。

### 自动模式

自动模式由任务台调度：先读取币种资金分配，再根据任务周期、已启用策略和识别到的行情场景生成信号。切换前必须完成策略配置并手动启动任务；运行期间仍受全部资金与风控规则约束。

### 模拟与实盘

没有有效 API 时，系统以本地模拟方式记录订单；连接 API 后可能提交真实订单。请始终通过 API 状态确认当前环境。

## 11. 升级与数据保留

了解升级方式、个性化配置保存位置与迁移规则。

### 个性化策略会保留吗

会。策略、资金、风控、持仓和 API 配置保存在操作系统的用户数据目录，不放在可被安装包覆盖的应用目录中。升级后新版会继续加载原配置，并为新增字段补充默认值。

### 旧版本配置如何处理

首次启动新版时，系统会自动复制旧的 ~/.bn_trader_*.json 到稳定用户目录，保留旧文件不删除。再次保存设置时，还会留下上一版 .bak 备份。

### 升级前应该做什么

1. 停止自动交易并关闭应用。
2. 备份整个 BN-Trader 用户数据目录。
3. 安装新版后核对策略、资金、风控、持仓与 API。
4. 先用手动或模拟模式验证，再恢复自动交易。

> 提示：详细流程请阅读 docs/升级与数据迁移.md。当前版本没有应用内静默自动更新。

## 12. 常见问题

连接、数据、主题与安装问题的快速处理方法。

### API 显示未连接

检查 Key 是否完整、账户权限是否正确、本机时间是否同步，以及网络能否访问 Binance。Secret Key 通常只在创建时显示，请妥善保存。

### 为什么订单被拦截

查看弹窗与风控日志。常见原因包括可用资金不足、单笔仓位过高、触发日亏损限制、连续亏损或处于冷静期。

### 暗色主题没有保存

主题选择会写入本机设置并在下次启动时恢复。若未恢复，请检查应用是否有权限写入当前用户设置。

### macOS 提示无法验证开发者

自动构建的测试安装包目前未签名。内部测试可在确认来源可信后按系统提示处理；公开发布应使用 Developer ID 签名并完成 Apple 公证。

# BN-Trader 软件升级与客户数据迁移

适用版本：v1.0.0

定义桌面版本发布、客户配置持久化、旧版迁移、备份、回滚与安全处理规则。

> 当前状态：当前版本通过 GitHub Release 或安装包手动升级，尚未提供应用内自动下载与静默安装。

## 1. 设计目标

- 升级应用程序时，不要求客户重新设置策略、资金、风控、持仓和 API。
- 程序文件与客户数据物理分离，卸载或覆盖应用目录不会默认删除客户资料。
- 新版本增加参数时自动补充默认值，同时保留客户已经修改的值。
- 旧版本配置首次加载时自动迁移，并保留原文件用于人工回退。
- 设置保存失败时不破坏上一份可用配置。

## 2. 当前发布与升级方式

1. 开发版本合并到 main 后，自动同步生成产品资料、系统帮助和本升级文档。
2. 推送 v* 版本标签后，GitHub Actions 构建 Windows 和 macOS 安装包并创建 Release。
3. 客户关闭正在运行的 BN-Trader。
4. Windows 客户运行新版 NSIS 安装程序并覆盖原程序；macOS 客户用新版 App 替换应用程序。
5. 新版首次启动时加载稳定用户目录中的原配置，并自动补充新字段。

> 注意：当前没有应用内自动更新器，也不会后台静默替换程序。客户应只从可信的官方发布地址获取安装包。

## 3. 应用文件与客户数据的边界

- 应用目录：可执行文件、Qt 运行库、图标、帮助内容和程序代码。升级时允许覆盖。
- 偏好设置目录：策略、全局风险参数、API 配置和凭据环境文件。升级时必须保留。
- 业务数据目录：交易数据库和运行日志。正式安装包运行时位于系统用户数据目录。
- 系统偏好：明亮/暗色主题由 Qt QSettings 保存到当前操作系统的用户偏好存储。
- 自动化任务、参与投资的交易对和资金比例保存在稳定偏好目录，升级后继续沿用，但任务不会在重启后自动恢复运行。

## 4. 各平台稳定数据目录

- Windows 偏好设置：%APPDATA%\BN-Trader
- Windows 数据库与日志：%APPDATA%\BN-Trader
- macOS 偏好设置：~/Library/Application Support/BN-Trader
- macOS 数据库与日志：~/Library/Application Support/BN-Trader
- Linux 偏好设置：${XDG_CONFIG_HOME:-~/.config}/BN-Trader
- Linux 数据库与日志：${XDG_DATA_HOME:-~/.local/share}/BN-Trader

> 注意：源码开发模式的数据库和日志仍可位于项目工作目录；发布安装包使用系统用户数据目录。

## 5. 持久化文件说明

- strategies.json：五类场景策略的启用状态与个性化参数。
- global.json：资金、单笔风险、日度风险、账户风险和持仓参数。
- api.json：API Key、Secret Key 与环境标记。
- automation_tasks.json：自动化任务、币种资金分配比例与启用状态；升级后保留，启动状态不会持久化。
- .env：供运行时客户端加载的 API 环境变量。
- bn_trader.db：交易和业务数据数据库。
- bn_trader.log：运行日志。
- *.json.bak：客户再次保存设置时产生的上一版本备份。

## 6. 旧版本自动迁移

旧版分别使用 ~/.bn_trader_strategies.json、~/.bn_trader_global.json 和 ~/.bn_trader_api.json。新版在目标文件不存在时查找这些旧文件，将第一份有效配置复制到稳定目录，并保留旧文件不删除。旧应用目录中的 .env 也会在稳定凭据文件不存在时复制到用户目录。

> 注意：自动迁移采用复制而非移动，避免迁移中断后客户无法回到旧版本。

## 7. 配置结构升级

- 设置文件包含 _schema_version，用于未来识别配置结构版本。
- _schema_version 是配置元数据，不作为策略对象处理；策略加载器会忽略这类非策略顶层字段。
- 参数 schema v2 统一百分比单位：界面一律显示 0–100 的百分数，磁盘和运行时一律保存 0–1 的小数比例。
- 旧版策略文件中 0.7 表示 0.7% 的字段会迁移为内部值 0.007；旧版全局设置中的混合单位也会按明确字段表迁移，不再根据数值是否大于 1 猜测。
- 新版本读取旧配置后，会递归补充新增的默认字段。
- 客户已有字段优先，不会因为默认值变化而被静默覆盖。
- 删除或改变含义的字段必须在未来版本中提供显式迁移函数，不能直接复用旧字段名。
- 交易逻辑发生重大变化时，应在发布说明中指出受影响参数并要求客户复核。

## 8. 原子保存与备份

保存设置时，系统先在同一目录写入临时文件并刷新到磁盘，再通过原子替换更新正式文件。若正式文件已经存在，会先复制为 .bak。这样可以降低断电、磁盘错误或程序中断造成半写文件的风险。

1. 恢复单个设置时，先关闭 BN-Trader。
2. 复制当前 JSON 文件到其他位置留档。
3. 确认对应的 .json.bak 内容有效。
4. 用备份文件替换正式 JSON 文件。
5. 重新启动并在界面中核对参数。

## 9. API 凭据安全边界

API 配置为了跨版本保留，目前存放在当前操作系统用户目录中。Unix 类系统会尽量设置为仅当前用户可读写，但当前版本尚未接入 Windows Credential Manager 或 macOS Keychain，因此不能把本地明文存储视为硬件级安全保护。

- 只授予必要的读取或现货交易权限，绝不启用提现权限。
- 使用操作系统账户密码和磁盘加密保护本机。
- 不要把用户数据目录同步到公开网盘、代码仓库或共享目录。
- 怀疑凭据泄露时，应立即在交易所撤销 Key，而不是只删除本地文件。

## 10. 升级前客户检查清单

1. 确认当前没有待提交订单或正在执行的自动交易任务。
2. 记录当前版本号，并关闭应用。
3. 备份整个 BN-Trader 用户数据目录。
4. 从可信发布地址下载安装包，核对版本与文件来源。
5. 安装后先检查策略、资金、风控、持仓、API 和主题设置。
6. 先以手动或模拟模式验证行情和下单链路，再恢复自动交易。

## 11. 发布方升级验证清单

1. 更新 Config.APP_VERSION，并生成三份产品文档。
2. 运行全部单元测试和设置迁移测试。
3. 使用包含旧配置的测试账户启动新版，确认客户值保留且新字段已补齐。
4. 分别验证 Windows 安装覆盖与 macOS App 替换。
5. 验证升级后的数据库、日志、主题和 API 加载。
6. 验证明亮/暗色主题下的弹窗、确认框和输入控件均能正确切换。
7. 使用高价币和 USDC/USDT 验证 K 线价格精度、右侧价格轴与实时价标记。
8. 验证自动模式下右侧下单区正确锁定并显示原因，手动模式恢复可操作。
9. 验证左侧“研报”入口显示“建设中”，顶部 API 入口仍可正常打开。
10. 在 Release Notes 中列出参数语义、数据结构和交易行为变化。
11. 出现不兼容变更时提供回滚步骤与最低支持版本。

## 12. 回滚原则

程序回滚与数据回滚应分开处理。仅替换为旧版程序并不保证旧版能够理解新版数据结构。回滚前必须备份当前数据，并根据 Release Notes 判断是否需要恢复对应版本的 JSON 备份或数据库备份。

> 注意：当前版本为配置文件提供单级 .bak 备份，但尚未提供数据库自动版本快照和一键回滚界面。正式大版本升级前应人工备份整个用户数据目录。
